Ejercicio 6: Introducción a Eventos — El Contador Interactivo

Enunciado y Objetivos

Una interfaz gráfica estática carece de utilidad operativa si no es capaz de reaccionar ante los estímulos del usuario. En este ejercicio aprenderemos a capturar las pulsaciones del ratón aplicando el patrón **Delegación de Eventos** de Java, utilizando un único objeto controlador centralizado para gestionar múltiples botones simultáneamente.

Requisitos del programa:

  • Diseñar una ventana interactiva estructurada con una distribución elástica de tipo FlowLayout.
  • Añadir de forma secuencial tres componentes fijos: un botón de decremento ("-1"), una etiqueta numérica inicializada en "0" y un botón de incremento ("+1").
  • Enlazar ambos botones al mismo escuchador de eventos lógicos de la clase actual (this).
  • Ampliación: Evaluar dinámicamente dentro del método de callback qué botón físico ha disparado la acción mediante la función e.getSource().
---

Solución Completa

Hacemos que nuestra propia clase contenedora extienda de Frame e implemente formalmente el contrato de la interfaz ActionListener. Además, acoplamos un adaptador de ventana (WindowAdapter) para habilitar el cierre físico del programa al pulsar la cruz del marco.

package Tema11.Ejercicios.ejercicio6;

import java.awt.*;
import java.awt.event.*;

public class Ejercicio6 extends Frame implements ActionListener {

    // Declarar los componentes y variables de estado a nivel de clase (ámbito global)
    private Button btnRestar;
    private Button btnSumar;
    private Label lblNumero;
    private int contador;

    // Constructor encargado de ensamblar e inicializar el lienzo gráfico
    public Ejercicio6() {
        super("Contador Interactivo");
        this.contador = 0; // Estado inicial del modelo de datos

        // Configurar la distribución secuencial fluida
        setLayout(new FlowLayout());
        setSize(300, 100);

        // Instanciar los elementos gráficos
        btnRestar = new Button("-1");
        lblNumero = new Label("0", Label.CENTER);
        btnSumar = new Button("+1");

        // REGISTRO DE EVENTOS: Conectamos los botones al escuchador centralizado (this)
        btnRestar.addActionListener(this);
        btnSumar.addActionListener(this);

        // Añadir componentes respetando el orden secuencial del enunciado
        add(btnRestar);
        add(lblNumero);
        add(btnSumar);

        // CONTROL EXTRA: Evento utilitario para cerrar el proceso de la JVM al pulsar la "X"
        addWindowListener(new WindowAdapter() {
            @Override
            public void windowClosing(WindowEvent e) {
                System.exit(0);
            }
        });

        // Mostrar la ventana terminada al usuario
        setVisible(true);
    }

    // MÉTODO INTERCEPTOR (Callback): Se dispara automáticamente al interactuar con un botón
    @Override
    public void actionPerformed(ActionEvent e) {
        // e.getSource() devuelve la referencia en memoria del objeto exacto que provocó el evento
        if (e.getSource() == btnRestar) {
            contador--; // Lógica de decremento
        } else if (e.getSource() == btnSumar) {
            contador++; // Lógica de incremento
        }

        // Sincronizamos la vista mutando el int a String de forma segura
        lblNumero.setText(String.valueOf(contador));
    }

    // Método de arranque de la aplicación
    public static void main(String[] args) {
        // Instanciamos el constructor anónimo para inicializar el hilo de la interfaz
        new Ejercicio6();
    }
}

Análisis Técnico de la Delegación de Eventos

Comprendiendo el flujo interno de las señales lógicas en Java:

  • El Contrato ActionListener: Es una interfaz abstracta que exige la redefinición obligatoria del método actionPerformed(ActionEvent e). Actúa como un "interceptor" que permanece a la escucha esperando a que el sistema operativo capture un clic del usuario sobre el componente registrado.
  • ¿Por qué pasamos el argumento `this`? Al invocar btnSumar.addActionListener(this), le estamos indicando al botón que el objeto encargado de procesar su evento es la propia instancia de la clase Ejercicio6 en la que se encuentra integrado, unificando el control.
  • La magia de e.getSource(): Cuando compartimos un mismo escuchador para múltiples emisores, necesitamos un mecanismo de discriminación. El objeto ActionEvent porta metadatos del suceso. Al compararlo directamente mediante un condicional de identidad (==) frente a las variables globales btnRestar o btnSumar, aislamos quirúrgicamente la lógica de negocio sin necesidad de instanciar clases anónimas independientes para cada botón, ahorrando memoria y simplificando la legibilidad del código fuente.
