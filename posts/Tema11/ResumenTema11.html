Resumen de la Unidad 11: Interfaces Gráficas de Usuario (AWT)

En esta unidad didáctica hemos abandonado el paradigma tradicional de las aplicaciones en línea de comandos (CUI) para adentrarnos en el desarrollo de Interfaces Gráficas de Usuario (GUI). Hemos aprendido cómo estructurar aplicaciones de escritorio interactivas utilizando la biblioteca estándar y nativa de Java: AWT (Abstract Window Toolkit).

1. Arquitectura y Jerarquía de Componentes

En el ecosistema AWT, cualquier elemento visual que se renderiza en la pantalla se denomina genéricamente Componente (clase Component). Los componentes se dividen en dos categorías lógicas esenciales:

  • Contenedores (Containers): Componentes de infraestructura diseñados exclusivamente para albergar y organizar otros subcomponentes dentro de sí mismos (como Frame, Panel, o Dialog).
  • Controles (Controls): Componentes interactivos u hojas encargados de interactuar con el usuario o realizar acciones concretas (como Button, TextField, o Label).
Componente AWT Descripción Técnica y Propósito
Frame Contenedor de ventana independiente de nivel superior provisto de barra de título, bordes y botones nativos del sistema operativo.
Panel Contenedor subordinado invisible utilizado para segmentar, agrupar o aplicar sub-diseños de layouts dentro de una ventana.
Dialog Ventana emergente secundaria flotante, frecuentemente parametrizada como modal para bloquear la interacción de fondo.
TextField Campo de control que habilita la inserción e introducción de texto editable en una sola línea.

2. Modos de Distribución: Layout Managers

Para posicionar y ajustar las dimensiones de los elementos dentro de las ventanas de forma elástica, responsiva y multiplataforma, Java delega la tarea en los Layout Managers. Disponemos de dos metodologías de maquetación:

  • Posicionamiento Absoluto (Manual): Forzado mediante la llamada setLayout(null). Obliga a definir de forma milimétrica la posición fija de cada control mediante coordenadas de píxeles con el método setBounds(X, Y, Ancho, Alto). Es rígido y propenso a desconfigurarse según la plataforma.
  • Distribución Automática (Gestores): Implementaciones de software que adaptan dinámicamente la geometría de la interfaz:
    • FlowLayout: Organiza los componentes horizontalmente en una sola fila respetando su tamaño, saltando de línea automáticamente cuando no hay espacio físico.
    • BorderLayout: Divide la pantalla en cuatro regiones cardinales (Norte, Sur, Este, Oeste) y un área Central elástica que absorbe todo el espacio restante.
    • GridLayout: Organiza la interfaz en una cuadrícula simétrica rígida de filas y columnas idénticas, estirando los componentes uniformemente.
    • CardLayout: Estructura los componentes en forma de pila de cartas, donde solo un lienzo permanece visible y se alternan de manera dinámica.
---

3. Arquitectura Jerárquica de Menús

Los menús superiores de AWT no heredan de la clase Component común, sino de MenuComponent, estructurándose de forma obligatoria de abajo hacia arriba:

  1. MenuItem: Las opciones finales clicables que disparan las acciones de negocio (ej: "Guardar", "Salir").
  2. Menu: Las pestañas o bloques desplegables principales que cuelgan verticalmente al hacer clic sobre ellos (ej: "Archivo", "Color"). Almacenan los MenuItem.
  3. MenuBar: La barra o franja horizontal física de control superior anclada fijamente debajo de la barra de título. Alberga las pestañas de los menús y se vincula mediante frame.setMenuBar().
---

4. El Modelo de Gestión de Eventos

Un evento es un suceso que ocurre durante la ejecución del programa (como un clic del ratón, presionar una tecla o cerrar una ventana). Java implementa un mecanismo estructurado de Delegación de Eventos (basado en el patrón de diseño Observer), compuesto por tres actores fundamentales:

  1. La Fuente (Source): El componente gráfico emisor que sufre la acción y origina el suceso (ej: un Button).
  2. El Objeto Evento (Event): La instancia de datos (como ActionEvent o WindowEvent) que encapsula de forma segura los metadatos de la acción ocurrida.
  3. El Oyente (Listener): La interfaz de escucha (como ActionListener) encargada de implementar el método callback de resolución (actionPerformed()) para interceptar el evento.

Para dotar de vida a la aplicación, registramos el oyente en el objeto fuente mediante la llamada addXXXListener(). Dentro del controlador, podemos unificar escuchadores sobre múltiples botones utilizando el método discriminador e.getSource() para evaluar dinámicamente la referencia en memoria del componente disparador.
