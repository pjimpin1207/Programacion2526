Ejercicio 15: El Adivino del XML — Parseo y Árboles DOM

Enunciado y Contexto

Los archivos de configuración en formato XML son ampliamente utilizados en entornos de software. Para poder manipularlos desde Java, necesitamos procesar la cadena secuencial del archivo y transformarla en un árbol jerárquico de nodos en memoria. Para ello, emplearemos la API estándar de análisis DOM (Document Object Model).

Estructura del archivo base (config.xml):

Antes de ejecutar la aplicación, es necesario crear un documento llamado config.xml en la raíz del proyecto con las siguientes directivas de configuración de ejemplo:

<?xml version="1.0" encoding="UTF-8"?>
<configuracion>
    <base-datos>
        <host>localhost</host>
        <puerto>5432</puerto>
        <usuario>admin</usuario>
    </base-datos>
</configuracion>
---

Solución Completa

Implementamos las factorías de análisis estructural DocumentBuilderFactory para mapear las etiquetas y aplicamos un algoritmo de filtrado por tipo de nodo para aislar la información útil de los elementos.

package Tema9.Ejercicios.ejercicio15;

import org.w3c.dom.Document;
import org.w3c.dom.Element;
import org.w3c.dom.Node;
import org.w3c.dom.NodeList;
import javax.xml.parsers.DocumentBuilder;
import javax.xml.parsers.DocumentBuilderFactory;
import java.io.File;

public class Ejercicio15 {
    public static void main(String[] args) {
        try {
            // 1. Crear la factoría y el constructor de documentos estructurados
            DocumentBuilderFactory factory = DocumentBuilderFactory.newInstance();
            DocumentBuilder builder = factory.newDocumentBuilder();

            // 2. Parsear el archivo XML físico cargándolo en la memoria interna
            File archivo = new File("config.xml");
            Document documento = builder.parse(archivo);

            // Normaliza el árbol XML (elimina nodos de texto vacíos adyacentes de formateo)
            documento.getDocumentElement().normalize();

            // 3. Obtener el elemento raíz (<configuracion>)
            Element raiz = documento.getDocumentElement();
            System.out.println("Elemento raíz detectado: " + raiz.getNodeName());

            // 4. Obtener los hijos lógicos del nodo de configuración <base-datos>
            NodeList listaBaseDatos = documento.getElementsByTagName("base-datos");
            // Tomamos el primer bloque contenedor que aparezca (índice 0)
            Node nodoBaseDatos = listaBaseDatos.item(0);

            // 5. Extraer y evaluar los hijos directos del bloque contenedor
            NodeList hijos = nodoBaseDatos.getChildNodes();
            System.out.println("\nNúmero total de hijos detectados en el búfer: " + hijos.getLength());

            System.out.println("--- Listando hijos del Árbol ---");
            for (int i = 0; i < hijos.getLength(); i++) {
                Node hijo = hijos.item(i);

                // FILTRADO CRÍTICO: Evaluamos si el nodo inspeccionado es una etiqueta real
                if (hijo.getNodeType() == Node.ELEMENT_NODE) {
                    Element elemento = (Element) hijo; // Casting de Node general a Element
                    System.out.println("ELEMENTO REAL -> Nombre: " + elemento.getNodeName() +
                            " | Valor: " + elemento.getTextContent());
                } else {
                    // Muestra los nodos fantasmas (espacios en blanco, tabulaciones y saltos de línea)
                    System.out.println("Hijo " + i + " -> Tipo: " + hijo.getNodeType() +
                            " (TEXT_NODE Oculto) | Nombre: " + hijo.getNodeName());
                }
            }

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}

Preguntas de Reflexión y Análisis Técnico

¿Cuántos hijos detecta realmente el programa en el bloque y por qué?

Al ejecutar el código sin filtrar, descubriremos que el método getChildNodes().getLength() devuelve un número de hijos significativamente mayor al número de etiquetas reales que escribimos (detecta 7 hijos en lugar de las 3 etiquetas de configuración de base de datos).

Esto ocurre porque el analizador DOM de Java interpreta absolutamente todo lo que está dentro del archivo:

  • Cada salto de línea al terminar de escribir una directiva y cada espacio en blanco o tabulación que usamos para sangrar (indenter) el código es considerado por la API como un nodo de texto fantasma de tipo implícito bajo el nombre de #text (Node.TEXT_NODE).
  • Los verdaderos elementos: Los únicos nodos operativos de utilidad real para la lógica de negocio son aquellos cuyo tipo coincide exactamente con la constante estructurada de validación Node.ELEMENT_NODE (como <host>, <puerto> y <usuario>).
  • Por este motivo, siempre que se recorra un documento mediante un analizador de tipo jerárquico tradicional, es imperativo establecer la condicional de control de tipo de nodo antes de extraer los datos mediante getTextContent().
