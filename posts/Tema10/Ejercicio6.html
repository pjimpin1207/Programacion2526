Ejercicio 6: Modificación de Datos (Dirty Checking y Sentencias UPDATE en JPQL)

Enunciado

El objetivo de esta práctica es dominar los dos mecanismos fundamentales que ofrece la API de JPA para mutar o actualizar información en la base de datos: la sincronización implícita por estados de memoria y la ejecución explícita de comandos de actualización masiva mediante el lenguaje de consultas orientado a objetos JPQL.

Misiones requeridas:

  • Modificar el nombre del primer concierto (ID 10) para transformarlo a mayúsculas utilizando el acceso directo por identificador y mutación de propiedades (setters).
  • Modificar el nombre del segundo concierto (ID 20) a mayúsculas empleando una Query JPQL Dinámica de tipo UPDATE.
  • Realizar la misma acción sobre el segundo concierto pero valiéndose de una Query JPQL Estática precargada con la anotación @NamedQuery.

Estructura de la Solución

1. La Entidad con Grupo de Consultas Nombradas (EventoMusical)

Hacemos uso de la anotación contenedora @NamedQueries para agrupar de manera limpia múltiples definiciones estáticas. Observa la estructura del comando de actualización: operamos directamente sobre la clase y sus atributos.

package Tema10.Ejercicios.ejercicio6;

import Tema10.Ejercicios.ejercicio2.Artista;
import Tema10.Ejercicios.ejercicio2.GeneroMusical;

import javax.persistence.*;
import java.io.Serializable;
import java.math.BigDecimal;
import java.time.LocalDate;
import java.util.ArrayList;
import java.util.List;

@Entity
@NamedQueries({
        @NamedQuery(name = "buscarEventoPorIdEstatico", query = "SELECT e FROM EventoMusical e WHERE e.id = :idParam"),
        @NamedQuery(name = "actualizarNombreEstatico", query = "UPDATE EventoMusical e SET e.nombre = :nombreMayus WHERE e.id = :idParam")
})
public class EventoMusical implements Serializable {
    private static final long serialVersionUID = 1L;

    @Id
    @GeneratedValue(generator = "SecuenciaEj6", strategy = GenerationType.SEQUENCE)
    @SequenceGenerator(name = "SecuenciaEj6", allocationSize = 10)
    private Long id;

    private String nombre;
    @Temporal(TemporalType.DATE)
    private LocalDate fecha;
    private BigDecimal recaudacion;
    private GeneroMusical genero;
    @ElementCollection
    private List<Artista> artistasConfirmados = new ArrayList<>();

    private transient int control;

    public EventoMusical() {
    }

    public EventoMusical(Long id, String nombre, LocalDate fecha, BigDecimal recaudacion, GeneroMusical genero, int control) {
        this.id = id;
        this.nombre = nombre;
        this.fecha = fecha;
        this.recaudacion = recaudacion;
        this.genero = genero;
        this.control = control;
    }

    // Getters y Setters
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }

    public String getNombre() { return nombre; }
    public void setNombre(String nombre) { this.nombre = nombre; }

    public LocalDate getFecha() { return fecha; }
    public void setFecha(LocalDate fecha) { this.fecha = fecha; }

    public BigDecimal getRecaudacion() { return recaudacion; }
    public void setRecaudacion(BigDecimal recaudacion) { this.recaudacion = recaudacion; }

    public GeneroMusical getGenero() { return genero; }
    public void setGenero(GeneroMusical genero) { this.genero = genero; }

    public List<Artista> getArtistasConfirmados() { return artistasConfirmados; }
    public void setArtistasConfirmados(List<Artista> artistasConfirmados) { this.artistasConfirmados = artistasConfirmados; }

    public int getControl() { return control; }
    public void setControl(int control) { this.control = control; }

    public void addArtista(Artista artista) {
        this.artistasConfirmados.add(artista);
    }

    @Override
    public String toString() {
        return "EventoMusical [ID (Incrementos de 10)=" + id + ", Nombre=" + nombre + ", Género=" + genero +
                ", Fecha=" + fecha + ", Recaudación=" + recaudacion + "€ | (Variable interna 'control'=" + control + ")]";
    }
}

2. Programa de Modificación de Lotes (ActualizarEventos)

A diferencia de las consultas de lectura pura del tema anterior, todas las operaciones que alteran el estado físico de los datos requieren una transacción activa (begin() / commit()). Para disparar las consultas de modificación masiva en JPQL empleamos de forma obligatoria el método executeUpdate().

package Tema10.Ejercicios.ejercicio6;

import javax.persistence.EntityManager;
import javax.persistence.EntityManagerFactory;
import javax.persistence.Persistence;
import javax.persistence.Query;

public class ActualizarEventos {
    public static void main(String[] args) {
        // Conexión segura al almacén de objetos de ObjectDB
        EntityManagerFactory emf = Persistence.createEntityManagerFactory("objectdb:$objectdb/db/eventos.odb");
        EntityManager em = emf.createEntityManager();

        // FASE 1: Modificación implícita del primer evento mediante el Identificador
        em.getTransaction().begin();
        EventoMusical primerEvento = em.find(EventoMusical.class, 10L);
        // Al alterar el atributo de una entidad gestionada dentro de una transacción, JPA detecta el cambio automáticamente
        primerEvento.setNombre(primerEvento.getNombre().toUpperCase());
        em.getTransaction().commit();
        System.out.println("Primer evento actualizado mediante identificador (Sincronización Automática).");

        // FASE 2: Modificación del segundo evento utilizando JPQL Dinámico
        em.getTransaction().begin();
        String nombreActual = em.find(EventoMusical.class, 20L).getNombre().toUpperCase();

        // Ensamblado e inyección de parámetros en la sentencia dinámica de actualización
        Query queryDinamica = em.createQuery("UPDATE EventoMusical e SET e.nombre = :nombreMayus WHERE e.id = 20");
        queryDinamica.setParameter("nombreMayus", nombreActual);
        queryDinamica.executeUpdate(); // IMPORTANTE: Ejecuta la sentencia bulk de actualización
        em.getTransaction().commit();
        System.out.println("Segundo evento actualizado mediante JPQL Dinámica.");

        // FASE 3: Modificación del segundo evento utilizando JPQL Estático (NamedQuery)
        em.getTransaction().begin();
        Query queryEstatica = em.createNamedQuery("actualizarNombreEstatico");
        queryEstatica.setParameter("idParam", 20L);
        queryEstatica.setParameter("nombreMayus", em.find(EventoMusical.class, 20L).getNombre().toUpperCase());
        queryEstatica.executeUpdate(); // Consolidación en lote
        em.getTransaction().commit();
        System.out.println("Segundo evento actualizado mediante JPQL Estática.");

        // Liberación absoluta de recursos
        em.close();
        emf.close();
    }
}

Análisis de Mecánicas Internas de JPA

Comprendiendo cómo viajan los datos mutados al almacenamiento físico:

  • El milagro del Dirty Checking (Fase 1): Observa que para el primer evento (ID 10) nunca invocamos a un método como em.update() o em.persist() tras cambiar su nombre. Esto se debe a que al extraer el objeto con em.find(), este queda en estado Managed (Gestionado) dentro del mapa de control del EntityManager. Al hacer commit(), el framework ejecuta una inspección de cambios (Dirty Checking) y, si detecta discrepancias entre la RAM y el disco, lanza de forma automática el comando SQL/ODB necesario para actualizarlo.
  • Consultas Bulk con executeUpdate() (Fases 2 y 3): Cuando operamos con sentencias directas de tipo UPDATE o DELETE en JPQL, el comportamiento cambia. No estamos modificando una instancia concreta en memoria, sino emitiendo una orden de mutación directa hacia la base de datos. Por este motivo, el estándar JPA exige invocar de forma estricta al método executeUpdate() (que devuelve un entero indicando el número de filas o registros modificados), fallando de forma abrupta si intentáramos invocar a getSingleResult() o getResultList().
  • El peligro de desincronización de la caché: Las consultas de modificación masiva aplicadas mediante executeUpdate() viajan de forma directa a la base de datos sin actualizar las instancias de objetos que ya estuvieran cargadas previamente en la memoria RAM del programa. Es una excelente regla de diseño limpiar el contexto de persistencia mediante em.clear() o refrescar las entidades con em.refresh() si se planea seguir operando con ellas tras un proceso masivo en lote.
