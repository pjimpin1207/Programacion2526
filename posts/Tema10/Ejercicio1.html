Ejercicio 1: Creación de un POJO

Enunciado

El objetivo de este ejercicio es diseñar un objeto de transferencia de datos puro o POJO (Plain Old Java Object) que represente un evento musical. Se deben seleccionar minuciosamente los tipos de datos más adecuados para garantizar la precisión aritmética del dinero y la integridad temporal de la fecha.

Requisitos:

  • Campos para almacenar: nombre del evento, fecha y recaudación económica.
  • Implementar un método de prueba main que instancie y audite un evento de ejemplo.
  • Responder a cuestiones teóricas sobre el uso de BigInteger y las clases clásicas de tiempo en Java.

Solución Completa

1. La Clase de Entidad (EventoMusical / EventoMusical)

Para la recaudación utilizamos BigDecimal debido a que los tipos de punto flotante primitivos (double o float) provocan errores de redondeo impredecibles en operaciones financieras. Para la fecha, optamos por la API moderna de tiempo LocalDate.

package Tema10.Ejercicios.EventoMusical;

import java.math.BigDecimal;
import java.time.LocalDate;

public class EventoMusical {
    private String nombre;
    private LocalDate fecha;
    private BigDecimal recaudacion;

    // Constructor vacío por defecto (Requisito POJO)
    public EventoMusical() {
    }

    // Constructor completo parametrizado
    public EventoMusical(String nombre, LocalDate fecha, BigDecimal recaudacion) {
        this.nombre = nombre;
        this.fecha = fecha;
        this.recaudacion = recaudacion;
    }

    // Getters y Setters
    public String getNombre() {
        return nombre;
    }

    public void setNombre(String nombre) {
        this.nombre = nombre;
    }

    public LocalDate getFecha() {
        return fecha;
    }

    public void setFecha(LocalDate fecha) {
        this.fecha = fecha;
    }

    public BigDecimal getRecaudacion() {
        return recaudacion;
    }

    public void setRecaudacion(BigDecimal recaudacion) {
        this.recaudacion = recaudacion;
    }

    @Override
    public String toString() {
        return "EventoMusical [Nombre: " + nombre + ", Fecha: " + fecha + ", Recaudación: " + recaudacion + "€]";
    }
}

2. Clase de Prueba (Main)

Instanciamos el concierto utilizando el constructor explícito y pasamos la recaudación como una cadena de texto al constructor de BigDecimal para evitar cualquier pérdida de precisión inicial.

package Tema10.Ejercicios.EventoMusical;

import java.math.BigDecimal;
import java.time.LocalDate;

public class Main {
    public static void main(String[] args) {
        // Creamos la instancia del evento con datos simulados precisos
        EventoMusical concierto = new EventoMusical("Puro Latino", LocalDate.of(2026, 7, 15), new BigDecimal("1250450.75"));

        // Mostramos el resultado formateado por la consola
        System.out.println(concierto);
    }
}

Cuestionario de Reflexión Teórica

1. ¿Cuándo conviene usar BigInteger en lugar de Integer o Long?
Conviene utilizar BigInteger cuando necesitemos almacenar o realizar operaciones aritméticas con números enteros que **superen los límites físicos** de los tipos primitivos de Java. Mientras que un long tiene un valor máximo de $2^{63}-1$ (aproximadamente 9 trillones), BigInteger no tiene un límite fijo y se expande dinámicamente según la memoria RAM disponible. Es muy utilizado en criptografía y computación científica.

2. ¿Para qué sirve cada uno de los tipos clásicos: Date, Time, Timestamp y Calendar?

  • Date (java.util.Date o java.sql.Date): Tradicionalmente representaba un instante específico en el tiempo con precisión de milisegundos. En su vertiente SQL, se limita únicamente a almacenar año, mes y día (sin hora).
  • Time (java.sql.Time): Diseñado exclusivamente para almacenar información de horas, minutos y segundos (el reloj), ignorando por completo la fecha del calendario.
  • Timestamp (java.sql.Timestamp): Es un envoltorio que extiende de java.util.Date adaptado para bases de datos relacionales, capaz de almacenar fecha y hora combinadas con una precisión extrema añadida de nanosegundos.
  • Calendar (java.util.Calendar): Clase abstracta introducida para subsanar los problemas de internacionalización de Date. Permite realizar operaciones aritméticas complejas con el tiempo (añadir días, retroceder meses, cambiar husos horarios) convirtiendo un instante en campos específicos de calendario.

Nota: Actualmente, se prefiere el uso de la API moderna java.time (LocalDate, LocalTime, LocalDateTime) introducida en Java 8 frente a estos tipos clásicos debido a su inmutabilidad y seguridad frente a hilos (Thread-Safety).
