Ejercicio 8: Primera Conexión Relacional mediante JDBC

Enunciado

Dejamos atrás la persistencia en bases de datos orientadas a objetos (ObjectDB) para adentrarnos en el estándar del mercado relacional. El objetivo de esta práctica es integrar el conector de sistema (driver) de MariaDB en nuestro entorno de desarrollo y crear un script básico de enlace capaz de verificar la comunicación con el servidor local o capturar de forma defensiva cualquier fallo en la autenticación.

Requisitos y pasos:

  • Descargar e incorporar las dependencias del conector o driver `.jar` de MariaDB en el Build Path del proyecto.
  • Implementar un programa que intente abrir una conexión utilizando el administrador de drivers genérico con las credenciales de acceso asignadas de la base de datos (tienda).
  • Si el enlace tiene éxito, mostrar por consola el mensaje informativo: “Conexión establecida” y liberar los descriptores abiertos.
  • Si el enlace es denegado o el servidor está caído, capturar el error mostrando de forma amigable el mensaje: “No se pudo conectar”.

Solución Completa

Para interactuar con motores SQL recurrimos al paquete estándar java.sql, utilizando la clase factoría DriverManager y construyendo una URI de conexión estructurada con el protocolo JDBC.

package Tema10.Ejercicios.ejercicio8;

import java.sql.*;

public class Ejercicio8 {
    public static void main(String[] args) {
        try {
            // 1. Establecer la conexión física utilizando la URI de conexión JDBC
            // Formato estándar: jdbc:motor://servidor:puerto/nombre_bd
            Connection con = DriverManager.getConnection("jdbc:mariadb://localhost:3306/tienda", "fran", "Fr4n");
            
            // 2. Crear un canal descriptor de sentencias SQL (Statement)
            Statement stmt = con.createStatement();

            // Si el flujo llega a esta línea sin saltar al bloque catch, la conexión ha sido un éxito
            System.out.println("Conexión establecida");

            // 3. LIBERACIÓN OBLIGATORIA: Cerramos primero el canal de sentencias y luego la conexión de red
            stmt.close();
            con.close();

        } catch (SQLException e) {
            // 4. CONTROL DEFENSIVO: Captura de errores de red, base de datos inexistente o credenciales inválidas
            System.err.println("No se pudo conectar");
        }
    }
}

Análisis de la Arquitectura JDBC

Comprendiendo la capa de abstracción de conectividad de Java:

  • ¿Qué es JDBC (Java Database Connectivity)? Es una API estándar incluida en el núcleo de Java que define una serie de interfaces abstractas comunes para operar con bases de datos. Gracias a esto, los métodos que utilizamos (como getConnection() o createStatement()) son exactamente iguales sin importar si nos conectamos a MariaDB, Oracle, MySQL o PostgreSQL.
  • El rol del Conector o Driver: Dado que cada fabricante de bases de datos utiliza su propio protocolo de comunicación binario propietario a nivel de red, necesitamos descargar un archivo `.jar` intermedio. El driver actúa como el "traductor" encargado de implementar de forma real las interfaces abstractas de JDBC para un motor específico.
  • Desglose de la URI JDBC: La cadena de conexión funciona de forma análoga a una URL web pero segmentada por propiedades fijas:
    • jdbc: Protocolo base de inicialización del subsistema.
    • mariadb: Subprotocolo que le indica a Java qué driver del mapa interno debe activar.
    • //localhost:3306/ Dirección IP o dominio del servidor de alojamiento seguido del puerto de escucha asignado por defecto a MariaDB/MySQL.
    • tienda: El catálogo o base de datos concreta sobre la que solicitamos privilegios de sesión.
  • Gestión de Excepciones (SQLException): Las operaciones de red sobre servidores externos son entornos propensos a fallos imprevistos fuera del control de nuestro programa. Es obligatorio por contrato envolver estas llamadas en bloques try-catch capturando la excepción chequeada SQLException para evitar el cierre abrupto de la aplicación.
