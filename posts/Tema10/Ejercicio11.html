Ejercicio 11: Consultas Flexibles (execute) y Optimización por Lotes (Batch Processing)

Enunciado

En esta práctica avanzada de JDBC trabajaremos con dos herramientas clave para mejorar la versatilidad y el rendimiento de nuestras aplicaciones relacionales:

Objetivos del reto:

  • Investigar y aplicar el método genérico execute() para extraer de forma condicional los productos cuyo país de origen sea 'Espana'.
  • Agrupar las consultas de inicialización del ejercicio anterior en un único lote de ejecución atómica (Batch).
  • Ejecutar un segundo lote de procesamiento secuencial para insertar un nuevo conjunto de productos (kiwi, perito y plátano) controlando la transaccionalidad manual.

Solución Completa

Desactivamos el modo autoCommit para tomar el control de la transacción y utilizamos los métodos addBatch() y executeBatch() para enviar las instrucciones agrupadas al servidor en un solo viaje de red.

package Tema10.Ejercicios.ejercicio11;

import java.sql.*;

public class Ejercicio11 {
    public static void main(String[] args) throws Exception {
        try {
            // 1. Establecer la conexión con el servidor relacional de MariaDB
            Connection con = DriverManager.getConnection("jdbc:mariadb://localhost:3306/tienda", "fran", "Fr4n");
            Statement stmt = con.createStatement();

            System.out.println("Conexión establecida");

            // 2. USO DE EXECUTE(): Método genérico que sirve para cualquier tipo de sentencia SQL
            boolean tieneResultSet = stmt.execute("SELECT * FROM producto WHERE pais = 'Espana'");

            // Si devuelve true, significa que la consulta generó un cursor de lectura (ResultSet)
            if (tieneResultSet) {
                ResultSet rs = stmt.getResultSet();
                System.out.println("\n--- Productos Españoles usando execute() ---");
                while (rs.next()) {
                    System.out.println("- " + rs.getString("nombre") + " (" + rs.getString("descripcion") + ")");
                }
                rs.close(); // Cierre del cursor intermedio
            }

            // 3. PRIMER LOTE (BATCH): Agrupación de consultas estructurales anteriores
            // Desactivamos el auto-commit para gestionar la transacción de forma manual y segura
            con.setAutoCommit(false);
            
            stmt.addBatch("DELETE FROM producto");
            stmt.addBatch("INSERT INTO producto VALUES ('1', 'manzana', 'Manzanas golden', 50, 'Francia')");
            stmt.addBatch("INSERT INTO producto VALUES ('2', 'pera', 'Peras conferencia', 20, 'Espana')");
            stmt.addBatch("INSERT INTO producto VALUES ('3', 'uva', 'Uvas groumet', 30, 'Espana')");

            // Enviamos todo el bloque consolidado en un único viaje de Entrada/Salida
            stmt.executeBatch();
            con.commit(); // Confirmamos los cambios de manera persistente en disco

            // 4. SEGUNDO LOTE (BATCH): Inserción del nuevo catálogo de productos solicitados
            stmt.addBatch("INSERT INTO producto VALUES ('4', 'kiwi', 'Zaspri gold', 120, 'Nueva Zelanda')");
            stmt.addBatch("INSERT INTO producto VALUES ('5', 'perito', 'Peritos de Tavizna', 18, 'Espana')");
            stmt.addBatch("INSERT INTO producto VALUES ('6', 'plátano', 'Plátano canario', 40, 'Espana')");

            // Ejecución y confirmación del segundo bloque de transacciones
            stmt.executeBatch();
            con.commit();

            System.out.println("\nTodos los lotes transaccionales se han procesado de manera correcta.");

            // 5. LIBERACIÓN: Cierre formal de descriptores operativos
            stmt.close();
            con.close();

        } catch (SQLException e) {
            System.err.println("Ocurrió un error en la transacción o enlace con MariaDB: " + e.getMessage());
        }
    }
}

Análisis de Conceptos Avanzados de JDBC

Comprendiendo la flexibilidad y la optimización de flujos:

  • El comportamiento del método execute(): A diferencia de executeQuery() (que solo acepta lecturas) y de executeUpdate() (que solo acepta modificaciones), el método execute() es polimórfico y puede procesar cualquier instrucción SQL. Su valor de retorno es un booleano:
    • Devuelve true si la sentencia es de tipo consulta (SELECT). Para extraer los datos, debemos invocar a continuación a stmt.getResultSet().
    • Devuelve false si la sentencia es de tipo actualización o inserción (DML). Para auditar las filas afectadas, invocaríamos a stmt.getUpdateCount().
  • Procesamiento por Lotes (Batch Processing): En los ejercicios previos, cada llamada a un método de ejecución obligaba a la aplicación a abrir una trama de red, enviarla al servidor de MariaDB, esperar la respuesta del motor y cerrar el canal de retorno. Este comportamiento ralentiza drásticamente las operaciones masivas. Al utilizar addBatch(), las sentencias se acumulan en un búfer local de la memoria RAM. Al invocar a executeBatch(), se empaquetan todas las instrucciones juntas y se envían en un único viaje de Entrada/Salida, minimizando la latencia de red de forma radical.
  • Control de Transaccionalidad (con.setAutoCommit(false)): El procesamiento por lotes exige apagar el modo de confirmación automática. Si lo dejáramos activado, el lote perdería su naturaleza atómica. Al gestionarlo de forma manual con con.commit(), aseguramos que o bien se aplican todas las inserciones del lote con éxito, o bien se descartan por completo si una de ellas falla, manteniendo la integridad referencial de nuestra base de datos.
